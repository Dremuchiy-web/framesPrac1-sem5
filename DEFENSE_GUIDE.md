## Шпаргалка к защите (что сделали и как объяснять)

### Что было реализовано (кратко)
- **Мини веб‑служба** на ASP.NET Core minimal API.
- **In-memory хранилище**: `ConcurrentDictionary<int, Item>` + `Interlocked.Increment` для id (не ломается при параллельных POST).
- **3 endpoint-а**:
  - `GET /api/items` — список (+ простые фильтры/сортировка параметрами).
  - `GET /api/items/{id}` — элемент по id или 404 в едином формате.
  - `POST /api/items` — создание или 400 в едином формате.
- **Конвейер middleware (pipeline)**:
  - `RequestIdMiddleware` — `X-Request-Id` в ответе + в `HttpContext.Items`.
  - `RequestTimingMiddleware` — замер времени и лог.
  - `RequestLoggingMiddleware` — лог запроса/ответа.
  - `ExceptionHandlingMiddleware` — перевод исключений в единый JSON ошибки.
- **Единый формат ошибки**: `ApiErrorResponse { code, message, requestId }`, и `requestId` совпадает с `X-Request-Id`.
- **Интеграционные тесты** через `WebApplicationFactory<Program>`.

### Как “проходит запрос” (как объяснить)
1) Запрос приходит в приложение.
2) `RequestIdMiddleware` назначает requestId (или берёт из заголовка) и гарантирует, что он будет в ответе.
3) `RequestTimingMiddleware` запускает таймер, пропускает дальше, затем пишет время выполнения в лог.
4) `RequestLoggingMiddleware` пишет старт запроса, пропускает дальше, затем пишет финал (status code и duration).
5) Endpoint выполняет доменную логику (валидация, репозиторий).
6) Если случилось исключение — `ExceptionHandlingMiddleware` ловит его и отдаёт аккуратный JSON в едином формате.

### Что преподаватель может спросить и что отвечать

#### Про middleware и порядок
- **Почему такой порядок?**
  - RequestId должен быть самым ранним, чтобы он был в логах и в любом ответе (включая ошибки).
  - Timing/Logging должны “оборачивать” всё ниже (включая endpoints).
  - ExceptionHandling должен быть после логирования старта, чтобы исключения превращались в одинаковые ответы, а requestId оставался тем же.

#### Про единый формат ошибок
- **Зачем нужен `requestId`?**
  - Чтобы связать ответ клиенту с конкретной записью в логах (корреляция).
  - Это упрощает диагностику: клиент даёт requestId → по нему ищем событие в логах.
- **Почему не просто `ProblemDetails`?**
  - Можно и `ProblemDetails`, но требование задания — единый формат с кодом/сообщением/requestId. Мы сделали свой простой контракт.

#### Про “код ошибки”
- **Что такое `ErrorCodes` и зачем?**
  - Клиенту удобнее анализировать стабильный “код”, чем парсить текст сообщения.
  - Сообщение — для человека, код — для логики/контрактов/тестов.

#### Про валидацию
- **Где валидация и почему там?**
  - В доменном слое (`ItemValidation.ValidateCreate`), чтобы правило не зависело от транспорта (HTTP).
  - Endpoint вызывает доменную проверку; при нарушении кидается `DomainValidationException`, которую middleware превращает в 400.

#### Про хранение в памяти
- **Почему без БД?**
  - По заданию нужно in-memory, чтобы увидеть pipeline без отвлечения на инфраструктуру.
- **Какие риски?**
  - Данные пропадут при рестарте, нет репликации/персистентности.
  - При росте нагрузки/масштаба — переход на внешнее хранилище (БД) и изменение `IItemRepository`.

#### Про параллельные запросы
- **Как защищались от гонок при создании?**
  - id генерируется через `Interlocked.Increment`.
  - коллекция — `ConcurrentDictionary`, добавление атомарное `TryAdd`.

#### Про тесты
- **Почему интеграционные, а не только unit?**
  - Нужно проверить весь конвейер: middleware + формат ошибок + заголовки + маршрутизация.
  - Unit‑тестами сложно подтвердить, что `X-Request-Id` и JSON ошибки реально приходят клиенту.

#### Про “экспериментальную часть”
- **Что именно измеряли/наблюдали?**
  - Время выполнения пишется для каждого запроса (`RequestTimingMiddleware`), можно сравнить разные сценарии (список/получение/создание/ошибка).
  - Наблюдение: вне зависимости от сценария, `requestId` всегда присутствует, а ошибки унифицированы.
